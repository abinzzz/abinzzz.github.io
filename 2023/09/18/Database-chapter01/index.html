chapter01:Introduction

1.1 Database-System Applications

Database(DB):

  • 是什么：
    • 相关数据的集合
    • 以文件形式存储
  • 管理数据的特点：
    • 有价值
    • 量大
    • 在同一时间，多个用户和应用程序访问

Database management system(DBMS)/Database system(DBS)

  • 组成:
    • DB
    • 访问DB中数据的程序
  • DBMS提供了存储和检索信息的环境：
    • 信息存储结构的定义
    • 数据操作机制
    • 数据安全机制

Database system(DBS)=DB+DBMS+Users/Administers

Database system(DBAS)=DB+DBMS+Application programs+Users/Administers


1.2 purpose of Database Systems

解决：

  • 数据冗杂和不一致性
  • 数据访问困难
  • 数据隔离：由于数据分散在各种文件中，并且文件可能具有不同的格式，编写新的应用程序以检索适当的数据是困难的。
  • 完整性问题：存储在数据库中的数据值必须满足某些类型的一致性约束。
  • 原子性问题：全发生或者全不发生
  • 多用户并发访问：允许多个用户同时更新数据
  • 安全问题:用户和权限的关系

1.3 数据视图

数据库系统是一组相关数据和一组程序的集合，这些程序允许
用户访问和修改这些数据。数据库系统的一个主要目的是
为用户提供数据的抽象视图。也就是说，系统隐藏了数据如何存储和维护的某些细节
。

1.3.1 数据模型

数据模型：DBS中对数据组织与管理方式的抽象描述，例如数据组织的语法语义定义和数据操作

数据模型分类：

  • Relational Model*
  • Entity-Relationship Model
  • Semi-structured Data Model
  • Object-Based Data Model

1.3.2 Relational Data Model

在关系模型中，数据以表的形式表示。每个表有多个
列，每个列有一个唯一的名称。表格的每一行表示
一条信息。

1.3.3 Data Abstraction

由于许多数据库系统用户没有经过计算机培训，开发人员通过几个级别的数据抽象向用户隐藏的复杂性，以简化用户与系统的交互

View level: 计算机用户看到一组隐藏 数据类型细节的应用程序。

1.3.4 Instances and Schemas

Model: 用于描述数据、数据关系、
数据语义和一致性约束的概念工具的集合。

Instance: 在特定时刻存储在数据库中的
信息集合,类比变量。

Schema: 数据库的总体设计,数据库中数据的结构描述，类比变量的值。

1.4 Database Languages

数据库系统提供了一种数据定义语言(data-definition language, DDL)来指定数据库
模式，以及一种数据操作语言(data-manipulation language, DML)来表示数据库查询和更新。

1.4.1 Data-Definition Language(DDL)

我们通过一组由称为数据定义语言(DDL)的特殊语言
表示的定义来指定数据库模式。DDL还用于指定数据的其他
属性。

存储在数据库中的数据值必须满足一定的一致性约束,最小开销进行测试的完整性约束为:

  • Domain Constraints：可能值的域必须与每个
    属性相关联
  • Referential Integrity：在某些情况下，我们希望确保一个值
    出现在一个给定属性集的关系中，也出现在另一个关系的属性集
    中。
  • Authorization：谁能访问

DDL语句的处理，就像任何其他编程语言的处理一样，
生成一些输出。DDL的输出放在数据字典
中，其中包含metadata，即关于data about data。数据字典
被认为是一种特殊类型的表，只能由数据库系统
本身(不是普通用户)访问和更新。

1.4.3 Data-Manipulation Language

数据操作语言(data-manipulation language, DML)是一种允许用户访问或操作
数据的语言，该数据由适当的数据模型组织而成。


分类：

  • Procedural DMLs： 要求用户指定需要什么数据以及如何获取这些数据。(relational algebra)
  • Declarative DMLs (also referred to as nonprocedural DMLs)(easier)： 要求用户指定需要什么数据，但不指定如何获取这些数据。(SQL)

1.4.5 Database Access from Application Programs

SQL不支持的操作：来自用户的输入、输出到显示器或通过网络通信。这样的计算和操作必须用host langugae编写，如C/ c++、Java或Python，并带有访问数据库中的数据的embedded SQL查询。

Application programs访问数据库：

  • 语言扩展以使用embedded SQL
  • 使用API，该接口可用于将DML和DDL语句发送到数据库并检索结果

Application programs：用于与数据库交互的程序。

1.5 Database Design

  • 保持data independence角度出发，根据data models定义数据规范形式，在view，logical，physical三层次
  • data abstraction
  • conception design,logical design(决定database schema),physical design(决定数据库的物理布局)
  • 构造DBS的E-R diagrams,external schema/logical schema,internal schema的集合
  • 得到conceptual DB，logical DB，physical DB

1.6 Database Engine

DBS的功能组件：

  • storage manager
  • query processor
  • transaction management

1.6.1 Storage Manager

Storage Manager提供了数据库中存储的低级数据与应用程序之间的接口 ，以及向系统提交的查询 。


Storage Manager：

  • 存储、检索和更新数据库中的数据。
  • 与文件管理器交互

Storage Manager组成：

  • authorization and intergrity manager:测试完整性约束的满足情况并检查用户访问数据的权限
  • transaction manager:尽管系统发生故障，数据库仍然保持一致(正确)的状态，并且并发事务的执行不会发生冲突。
  • file manager:管理磁盘存储空间的分配和用于表示存储在磁盘上的信息的数据结构。
  • buffer manager:负责从磁盘存储中获取数据到主内存中，并决定哪些数据要缓存在主内存中.

Storage Manager实现了几个数据结构，作为物理系统实现的一部分:

  • data files:存储数据库本身
  • data dictionary:存储有关DB结构的metadata，特别是DB的schema
  • indices:提供快速访问数据项,类似于索引。

1.6.2 Query Processor

Query Processor组件包括：

  • DDL interpreter:解释DDL语句并将定义记录在数据字典中
  • DML compiler:将查询语言中的DML语句转换为由查询计算引擎理解的低级指令组成的计算计划;执行查询优化
  • Query evaluation engine:执行 DML编译器生成的低级指令。

1.6.3 Transaction Management

atomicity:原子性，全有或无,例如借贷双方同时发生或者双方同时不发生
consistency：一致性，例如借贷双方的金钱总额是一定的。
durability：持久性，例如在成功执行资金转移后借贷双方新值必须保持不变，尽管存在系统故障的可能性

Transaction是在DB应用程序中执行单个逻辑功能
的操作的集合

Transaction Management组件：

  • concurrency-control manager：负责控制并发事务之间的交互
    ，以保证数据库的一致性。
  • recovery manager：须检测系统故障，恢复数据库到故障发生前的状态。

1.7 Database and Application Architecture

Centralized databases:a few cores,shared memory

parallel databases:

  • many core shared memory
  • shared disk
  • shared nothing

distributed databases:

  • geographical distribution
  • schema/data heterogeneity


two-tier architecture：
应用程序驻留在客户机机器上，并通过查询语言语句在服务器机器上调用DBS功能


three-tier architecture：

  • client machine仅充当前端，不包含任何直接的数据库调用
  • client end与application server通信
  • application server与DBS通信以访问数据

1.8 Database Users and Administrators

数据库系统的主要目标是从数据库中检索信息并在数据库中存储新的信息。

1.8.1 Database Users and User Interfaces

users分类：

  • sys admin
  • DBA
  • application user
  • database developer

1.8.2 Database Administrator

Database Administrator(DBA):对系统具有这种集中控制的人

DBA的功能：

  • Schema定义
  • 存储结构和访问方法定义
  • Schema和物理组织修改
  • 授予数据访问的authorization
  • 定期维护
  • 定期备份DB
  • 确保有足够的空闲磁盘空间用于正常运行
  • 根据需要升级磁盘空间
  • 监视在DB上运行的作业
