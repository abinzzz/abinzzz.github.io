Summary

  • A database-management system (DBMS) consists of a collection of interrelated data and a collection of programs to access those data. The data describe one particular enterprise.
  • The primary goal of a DBMS is to provide an environment that is both convenient and efficient for people to use in retrieving and storing information.

  • Database systems are ubiquitous today, and most people interact, either directly or indirectly, with databases many times every day.

  • Database systems are designed to store large bodies of information. The management of data involves both the definition of structures for the storage of information and the provision of mechanisms for the manipulation of information. In addition, the database system must provide for the safety of the information stored in the face of system crashes or attempts at unauthorized access. If data are to be shared among several users, the system must avoid possible anomalous results.

  • A major purpose of a database system is to provide users with an abstract view of the data. That is, the system hides certain details of how the data are stored and maintained.

  • Underlying the structure of a database is the data model : a collection of conceptual tools for describing data, data relationships, data semantics, and data constraints.

  • The relational data model is the most widely deployed model for storing data in databases. Other data models are the object-oriented model, the object-relational model, and semi-structured data models.

  • A data-manipulation language(DML)is a language that enables users to accessor manipulate data. Nonprocedural DMLs, which require a user to specify only what data are needed, without specifying exactly how to get those data, are widely used today.

  • A data-definition language(DDL)is a language for specifying the database schema and other properties of the data.

  • Database design mainly involves the design of the database schema. The entity- relationship (E-R) data model is a widely used model for database design. It provides a convenient graphical representation to view data, relationships, and con- straints.

  • A database system has several subsystems.

    • The storage manager subsystem provides the interface between the low-level data stored in the database and the application programs and queries submitted to the system.
    • The query processor subsystem compiles and executes DDL and DML statements.

  • Transaction management ensures that the database remains in a consistent (cor- rect) state despite system failures. The transaction manager ensures that concur- rent transaction executions proceed without conflicts.

  • The architecture of a database system is greatly influenced by the underlying computer system on which the database system runs. Database systems can be centralized, or parallel, involving multiple machines. Distributed databases span multiple geographically separated machines.
  • Database applications are typically broken up into a front-end part that runs at client machines and a part that runs at the backend. In two-tier architectures, the front end directly communicates with a database running at the back end. In three- tier architectures, the back end part is itself broken up into an application server and a database server.
  • There are four different types of database-system users, differentiated by the way they expect to interact with the system. Different types of user interfaces have been designed for the different types of users.


Review Terms

Database-management system (DBMS)：一组相互关联的数据和一组访问这些数据的程序。

数据库(Data Base，DB) 是按一定结构组织并长期存储在计算机内的、可共享的大量数据的有机集合。其实就是存放数据的仓库，只不过这些数据存在一定的关联、并按一定的格式存放在计算机上。例如，把一个学校的学生、课程、学生成绩等数据有序的组织并存放在计算机内，就可以构成一个数据库。

数据库管理系统（Data Base Management System，DBMS） 是管理和维护数据库的系统软件。常用的DBMS有：Oracle、DB2、SqlServer、MySql等

数据库管理员(Date Base Administrator ,DBA) 管理操作数据库人员。

数据库系统（Data Base System，DBS） 是实现有组织的、动态地存储大量关联数据、方便多用户访问的计算机软件、硬件和数据资源组成的系统，简化为：DBS=计算机系统（硬件、软件平台、人）+DBMS+DB

Database-system applications: 一种计算机程序，其主要目的是从计算机化的数据库中输入和检索信息。

Online transaction processing: 这指的是管理面向事务应用程序的系统。OLTP系统旨在支持大量的短在线交易，如订单输入、金融交易和客户记录管理。
Data analytics: 对数据进行处理以得出结论，并推断规则或决策过程，然后将其用于驱动业务决策。

File-processing systems： 这些系统将永久记录存储在各种文件中，并需要不同的应用程序将记录提取并添加到适当的文件中。在文件处理系统中，由于文件结构的不同以及不同文件中信息的重复，会导致数据冗余和不一致

Data inconsistency: 当同一数据的不同副本不一致时，就会发生这种情况。例如，如果一个学生的地址发生了变化，更新反映在一个部门的记录中，但没有反映在另一个部门中，这将导致数据不一致

Consistency constraints: 这些规则确保数据库中数据的完整性。它们确保存储在数据库中的数据值满足某些类型的约束。例如，一致性约束可能要求部门帐户的余额永远不能低于零

Data abstraction:这个概念有助于向用户隐藏数据库系统中数据结构的复杂性，并简化他们与系统的交互

  • Physical level:描述数据的实际存储方式，详细说明复杂的底层数据结构
  • Logical level:描述存储的数据，表示数据的类型及其关系，而不必担心它们的存储方式
  • View level:提供数据库的各种视图，每个视图针对不同的用户需求进行定制，允许用户仅访问数据库的一部分

Instance: 数据库上下文中的实例通常指数据库在某一时刻的特定状态

Schema: 数据库的总体设计

  • Physical schema: 物理模式描述了物理层上的数据库设计。物理模式是数据管理中的一个术语，用于描述如何使用特定的数据库管理系统(DBMS)在辅助存储中表示和存储数据(文件、索引)。
  • Logical schema:逻辑模式是独立于特定数据库管理产品或存储技术(物理数据模型)的特定问题领域的数据模型，但根据数据结构(如关系表和列、面向对象类或XML标记)表示
  • Subschema: 子模式是模式的子集，为某些用户或应用程序定义特定的视图或与数据库的交互机制

Physical data independence: 这个术语指的是在不改变逻辑模式的情况下改变物理模式的能力。它允许更改数据的存储方式，而不会影响数据的访问和管理方式。

Data models：

  • Entity-relationship model：该模型将数据表示为实体以及这些实体之间的关系。它广泛用于概念设计，其中实体表示现实世界的对象，关系表示这些对象如何相互作用。
  • Relational data model：该模型将数据组织成表(关系)，其中每个表由行(元组)和列(属性)组成。它以强大的数学基础而闻名，是SQL和大多数现代数据库的基础。
  • Semi-structured data model：该模型用于处理不能整齐地放入表、行和列中的数据。它很灵活，经常用于XML、JSON甚至电子邮件等数据，这些数据的结构可能不同或部分未知。
  • Object-based data model：该模型将面向对象的程序设计特性集成到数据库中。它将数据表示为对象，类似于面向对象编程，允许更复杂的结构和关系，在需要复杂数据表示的应用程序中很有用，如CAD或多媒体数据库。

Database languages

  • Data-definition language：这用于指定数据库模式和定义数据的附加属性。DDL允许创建、修改和删除数据库结构，如表、索引和约束。
  • Data-manipulation language：DML用于表达数据库查询和更新。它包括允许用户从数据库插入、更新、删除和检索数据的命令
    • Procedural DML：指定需要哪些数据以及如何获取数据
    • Declarative DML：指定需要哪些数据，而不显式说明如何获取数据。
    • nonprocedural DML：结合了Procedural和Declarative的特性。
  • Query language：它是DML的一个子集，专门为查询数据而设计。这允许用户通过指定标准从数据库检索特定信息
  • Data-definition language
    • Domain Constraints：定义关于某个属性的允许值的规则
    • Referential Integrity：确保表之间的关系保持一致
    • Authorization：包括向用户授予不同操作的权限，如读取、插入、更新和删除
      • Read authorization
      • Insert authorization
      • Update authorization
      • Delete authorization

Metadata： 元数据是关于数据的数据。它描述了数据库中存储的数据的结构和性质，例如表的结构、列的类型和约束等
Application program：在数据库环境中，应用程序指的是用于访问和操作数据库中数据的软件程序

Database design：

  • Conceptual design： 设计数据库的高级结构，通常使用实体-关系模型来表示。
  • Normalization：确保数据库的设计避免冗余和依赖问题，通常通过将数据组织成多个相关的表来实现。
  • Specification of functional requirements：确定数据库需要支持的操作和功能。
  • Physical-design phase：在这个阶段，设计者确定数据库的物理存储结构，如文件的组织形式和索引结构。

Database Engine：负责存储、查询、更新和管理数据库数据的核心服务和组件

  • Storage manager：负责数据库数据的存储和检索

    • Authorization and integrity manager：负责维护数据的安全性和完整性。
    • Transaction manager：监督数据库事务的执行，确保原子性、一致性、持久性，并管理并发和恢复
    • File manager：管理磁盘存储空间的分配以及用于表示存储在磁盘上的信息的数据结构
    • Buffer manager：负责从磁盘存储器中取出数据到主存，并将数据写回磁盘。它管理内存缓冲区并优化数据检索和存储过程
    • Data files：这些是存储数据库数据的磁盘上的实际文件。文件管理器通常处理这些文件
    • Data dictionary：关于数据库数据的信息存储库，包括模式定义、数据类型和约束。数据库管理系统使用它来保证数据的完整性和一致性
    • Indices：提高数据库上数据检索操作速度的数据结构，但代价是额外的写操作和维护它们所需的存储空间

  • Query processor：由几个组件组成，包括DDL解释器、DML编译器、查询优化器和查询求值引擎。它将用户查询和数据操作命令转换为用于数据检索和修改的低级指令

    • DDL interpreter：解释数据定义语言语句并在数据字典中记录定义
    • DML compiler：将数据操作语言语句转换为由查询求值引擎可以理解的低级指令组成的求值计划
    • Query optimization：涉及选择执行给定查询的最有效方法，通常由DML编译器完成
    • Query evaluation engine：执行由DML编译生成的低级指令

  • Transactions：作为单个逻辑工作单元执行的操作序列

    • Atomicity：确保事务要么全有，要么全无
    • Consistency：确保事务将数据库从一种有效状态转换为另一种有效状态
    • Durability：确保事务的结果被永久存储
    • Recovery manager：管理在发生故障后将数据库恢复到一致状态，从而确保持久性和原子性
    • Failure recovery：在发生故障后恢复数据库的过程，通常由恢复管理器处理
    • Concurrency-control manager：管理多个事务对数据库的并发访问，确保数据的一致性和完整性

Database Architecture：指数据库系统的结构

  • Centralized：在单个服务器上
  • Parallel：多个cpu一起工作
  • Distributed：数据分布在多个地理上独立的站点上

Database Application Architecture：应用程序使用数据库的设计

  • Two-tier：客户端和服务器

  • Three-tier：客户端、应用服务器和数据库服务器

  • Application server：

Database administrator (DBA)：负责管理数据库的人员，包括模式定义、存储结构、修改、访问授权和日常维护

1.7 List four significant differences between a file-processing system and a DBMS.

物理和逻辑访问的协调: DBMS协调对数据的物理和逻辑访问，而文件处理系统只协调物理访问。这意味着DBMS理解数据的结构，并且能够以比文件系统更抽象的方式处理数据，而文件系统只处理数据在存储介质上的位置。

数据复制: DBMS通过确保所有被授权访问的程序都可以使用单个数据块来减少数据复制。相反，一个程序在文件处理系统中写入的数据可能无法被另一个程序读取，从而导致潜在的数据冗余

灵活地访问数据: DBMS被设计为允许通过查询灵活地访问数据。另一方面，文件处理系统被设计为允许预先确定对数据的访问，通常是通过编译程序。这意味着dbms可以处理特殊查询，并提供多种查看和分析数据的方法，而文件系统仅限于程序开发人员预期的操作。

并发访问: DBMS是为协调多个用户同时访问相同数据而设计的，而文件处理系统通常是为一个或多个程序同时访问不同的数据文件而设计的。在文件处理系统中，只有当两个程序都对文件具有只读访问权限时，才能同时访问该文件。这证明了DBMS处理复杂场景的能力，其中多个用户或进程可能需要并发地读取或写入相同的数据，而不会导致冲突或数据损坏


1.8 解释物理数据独立性的概念及其在数据库系统中的重要性。

物理数据独立性是指不需要重写应用程序就能修改物理方案的能力。这些修改包括从非阻塞记录存储到阻塞记录存储，或者从顺序访问文件到随机访问文件。

逻辑数据独立性是指无需重写应用程序就能修改概念方案的能力。这样的修改可能是向记录添加一个字段，应用程序可以从程序中看到这种更改。

1.10 请列出至少两个原因，说明为什么数据库系统支持使用声明性查询语言(如SQL)进行数据操作，而不是仅仅提供一个C或C函数库来执行数据操作

抽象级别的提高：声明性查询语言允许用户指定他们想要什么（即所需的数据），而不是如何完成一个任务（即获取数据的具体步骤）。这种高级别的抽象允许用户专注于数据的逻辑结构而不是数据的物理布局，从而简化了数据库查询的编写和理解。

简化查询优化：使用声明性语言，数据库系统可以自动优化查询，以便高效地检索数据。系统可以决定数据访问的最佳路径，而无需用户干预。相比之下，使用程序库，优化负担通常落在了程序员身上，这要求他们具备深厚的专业知识，并可能导致非最优的数据访问路径。

跨平台兼容性：SQL作为标准化查询语言，具有跨平台的兼容性。它可以在多种数据库系统上运行而不需要修改，这为数据库操作提供了更大的灵活性和可移植性。而特定语言的函数库则通常依赖于特定的平台或数据库系统。

易于学习和使用：SQL的语法相对简单直观，易于学习，这使得非程序员也能够编写和理解查询，从而使得更广泛的用户群体能够利用数据库系统。

安全性和访问控制：数据库系统通过SQL提供了细粒度的安全性和访问控制。管理员可以精确地控制哪些用户可以执行哪些SQL操作，这种控制通常比直接使用程序库更细致和更易于管理。

维护和可扩展性：由于SQL是高度抽象的，当底层数据库结构变化时，通常不需要修改SQL查询。这与使用程序库相比，任何底层变化都可能需要更新和重新编译代码，导致维护困难和成本高昂。


1.11 假设两个学生想要注册一门只有一个名额的课程。数据库系统的什么组件可以防止两个学生都被分配到最后一个座位?

锁定机制（Locking Mechanisms）：数据库系统使用锁来控制对数据库对象（如行、页或表）的并发访问。在这种情况下，当第一个学生尝试注册课程时，数据库系统会在相应的数据库记录上放置一个锁。只有当这个学生的事务完成（提交或回滚）后，锁才会释放，其他学生才能尝试注册。

多版本并发控制（Multiversion Concurrency Control, MVCC）：这是另一种并发控制技术，它允许多个事务看到同一数据的不同版本。在注册课程的情况下，每个学生的事务可能会看到课程有空位的不同“版本”。然而，实际提交事务时，只有第一个成功的事务会被接受，第二个会因为冲突而失败。

事务隔离级别（Transaction Isolation Levels）：数据库通常提供不同的事务隔离级别，以控制一个事务所做的更改对其他并发运行的事务的可见性。较高的隔离级别可以防止并发事务中的问题，如脏读、不可重复读和幻读，这些都可能导致两个学生注册到同一个座位。

序列化（Serialization）：在最高的隔离级别下，事务是序列化的，这意味着每个事务都是依次执行的，就好像没有其他事务并发执行一样。虽然这可能会减少系统的并发性能，但它确保了数据的完整性。


1.12 解释两层和三层应用程序体系结构之间的区别。哪个更适合web应用程序?为什么?

两层架构，通常称为客户端/服务器架构，包括：

  • 客户端层：负责呈现用户界面和用户交互。在这层中，客户端直接与数据库服务器通信。
  • 服务器层：通常是一个数据库服务器，负责数据管理和查询处理。


三层架构则更加分离，包括：

  • 表示层：即用户界面层，负责与用户交互并显示数据。
  • 业务逻辑层（应用服务器）：处理应用程序的业务逻辑，例如计算、数据查询的逻辑处理，以及执行特定任务的业务规则。
  • 数据层：即数据库层，负责存储和管理数据。


对于Web应用程序来说，三层架构更为合适，原因包括：

  • 可伸缩性：三层架构允许每层独立地伸缩。随着用户数量的增加，你可以仅仅增强应用服务器层，而不必变动其他层。
  • 灵活性和维护性：业务逻辑独立于用户界面，这意味着后端的更改不会影响前端，反之亦然。这简化了应用程序的维护和更新。
  • 负载分散：在业务逻辑层和数据库层之间分开可以更有效地管理负载。例如，通过缓存策略和负载均衡可以提高应用程序的响应性和可用性。
  • 安全性：三层架构通过附加的业务逻辑层提供了更好的安全性，因为用户的请求不是直接发送到数据库，而是经过中间层的处理，可以在此实施安全策略。
  • 易于测试和部署：由于业务逻辑层是独立的，开发人员可以单独测试业务逻辑而无需关心数据存储和用户界面，这通常可以提高开发效率。
  • 技术多样性：三层架构允许在不同层使用最适合该层任务的技术，例如，在表示层使用JavaScript，在业务逻辑层使用Java或C#，在数据层使用SQL。


1.13 描述至少三个可能用于在社交网络系统(如Facebook)中存储信息的表

用户表（Users）：

  • UserID: 唯一标识每个用户的主键。
  • Username: 用户名，用于登录和标识。
  • PasswordHash: 用户密码的哈希值，出于安全考虑不直接存储密码。
  • Email: 用户的电子邮件地址。
  • ProfilePicture: 用户头像的链接或路径。
  • Bio: 用户的简介或生平。
  • BirthDate: 用户的出生日期。
  • JoinDate: 用户加入社交网络的日期。
  • Location: 用户的地理位置信息。


朋友关系表（Friendships）：

  • FriendshipID: 表中每条朋友关系的唯一标识。
  • UserID1: 参与朋友关系的第一个用户的ID。
  • UserID2: 参与朋友关系的第二个用户的ID。
  • Status: 朋友关系的状态（例如，请求中、已接受）。
  • StartDate: 成为朋友的日期。
  • ActionUserID: 发起朋友请求的用户的ID。


帖子表（Posts）：

  • PostID: 帖子的唯一标识。
  • UserID: 发布帖子的用户ID。
  • Content: 帖子的文本内容。
  • PostDate: 帖子发布的日期和时间。
  • Location: 帖子相关联的地理位置。
  • Visibility: 帖子的可见性设置（例如，公开、朋友、仅自己）。
  • LikesCount: 帖子获得的点赞数。
  • CommentsCount: 帖子的评论数。
