1. 软件架构

1.1 软件架构示意图

在Django的MTV架构中，模型（Model）是数据存储层，处理与数据库的交互和数据的增删改查；模板（Template）是表现层，负责定义如何显示数据；视图（View）则是业务逻辑层，处理具体的业务逻辑，并在模型和模板之间起到桥梁的作用

根据您上传的文件内容，我可以结合这些内容详细解释 Django 的模型-视图-模板（MVT）架构如何工作：

  1. 用户互动：用户通过浏览器与应用程序的前端进行交互，例如点击按钮、提交表单等。这些操作会向服务器发送 HTTP 请求。例如，在您的应用程序中，当用户想要改变空调设置时，他们会与 client-on.html 或 client-off.html 等 HTML 模板中定义的表单或按钮进行交互。

  2. 视图：在 Django 中，视图接收这些 HTTP 请求，并执行相应的业务逻辑。您的 views.py 包含了像 login 和 guanli_login 这样的函数，用于处理不同类型的用户请求。例如，当用户点击“高速”按钮以改变空调的风速时，一个请求被发送到一个视图函数（视图中的 change_high），然后该函数处理此请求并与模型交互以改变空调的状态。

  3. 模型：模型是应用程序数据的抽象表示，负责访问和存储数据（如空调状态、温度等）。它们直接与数据库进行交互，执行如查询、更新等操作。在您的 models.py 中，类似 Scheduler 和 Room 的类代表这些模型，封装了与数据库交互的逻辑。

  4. 模型获取/保存：视图通过模型层与数据库进行交互，模型层负责具体的数据库操作，比如更新房间温度或风速设置，然后将结果保存到数据库中。这在您的模型中很明显，数据库交互被封装在方法中。

  5. 数据回传给视图：数据处理完成后，模型将结果返回给视图。视图根据这些数据来准备最终要呈现给用户的 HTML 页面内容。

  6. 模板：模板定义了网页的结构和布局。它们使用 HTML 编写，并可以包含 Django 模板语言的标签和变量，用于插入动态数据。您的 HTML 文件，如 login.html、reception.html 和 report.html，作为模板，决定了用户界面的结构和显示方式。

  7. 渲染后的 Html 页面：视图使用上下文数据渲染模板，产生完整的 HTML 页面。Django 的模板引擎负责将模板中的变量替换为实际的数据，生成最终的页面内容。

  8. 返回渲染后的 Html 页面回给视图：渲染好的 HTML 页面被返回给视图函数。

  9. 用户得到回复的内容：最后，视图将渲染好的 HTML 页面作为 HTTP 响应发送给用户的浏览器，用户的浏览器接收到这个响应后，用户就可以看到更新后的页面。

此架构的合理性在于：

  • 分工明确：模型、视图、模板三者职责清晰分离，有助于代码的组织和维护。
  • 灵活性高：可以灵活地在视图中处理复杂的业务逻辑，而不会干扰到数据表示（模型）和页面展示（模板）。
  • 易于扩展：在此架构下，开发者可以方便地扩展模型、增加视图逻辑或修改模板，来应对应用的变化和增长。
  • 代码重用：模型可以被多个视图重用，模板可以通过继承和包含标签来重用页面的公共部分。

1.2 分层结构说明

在Django的MTV架构中，各层次的分工是这样的：

  1. 接收前端请求的对象：

    • URL分发器（urls.py）：当一个HTTP请求到达Django服务器时，Django首先会检查urls.py文件中定义的urlpatterns列表。Django使用正则表达式匹配请求的URL路径。对于每一个path函数，Django都会在其第一个参数中定义一个特定的URL模式。如果HTTP请求的URL与某个模式匹配，Django则会调用该模式对应的视图函数进行处理。例如，对于路径path(‘login/’, views.login)，当用户访问/login/这个URL时，Django会找到并执行views.py文件中定义的login视图函数。这个视图函数将处理请求，并返回一个HTML内容。

  2. 创建对象实例的对象：

    • 模型（models.py）：在Django中，模型负责定义数据结构，并提供了一个对象关系映射（ORM）层，这个ORM层可以创建新的实例（或记录）。当视图层需要创建新的数据记录时，它会调用模型层的相应方法来创建和初始化对象的实例。Room类代表房间，Scheduler类充当中控机，负责接收请求并根据请求分配服务对象，进行计费等功能。当Scheduler接收到一个新的房间请求时，它会调用request_on方法，该方法可能会创建一个新的Room对象实例，表示房间已经开机并准备接受服务。

  3. 处理请求具体要求的对象：

    • 视图（views.py）：视图负责处理用户的请求。它会调用模型层的方法来获取、更新或删除数据，并且决定哪个模板应当被用来显示数据。用户如果想开关机，视图首先通过 get_room_id 获取房间ID，然后根据房间的当前状态（开机或关机），它会调用 Scheduler 类的 request_on 或 request_off 方法来更新房间状态，并重定向到相应的页面。

  4. 数据同步和存储的对象：

    • 数据库（models.py）：虽然在您提供的图中没有直接显示数据库，但在Django架构中，数据库是用于数据同步和存储的主要对象。模型层通过ORM与数据库交互，进行数据的读写操作。
    • 模型（models.py）：模型层同样参与到数据的同步和存储中。它定义了数据的结构和行为，ORM使得开发者能够以编程的方式对数据库进行查询和更新操作，而不需要编写SQL代码。模型层确保了数据的一致性和有效性。
    • 例如客户调整了房间的目标温度,这个动作触发了一个前端请求，请求通过 Django 的 URL 调度器被传递到相应的视图函数.在这个函数中调用 ServingQueue 实例的 set_target_temp 方法，传入房间号和新的目标温度。set_target_temp 方法遍历 ServingQueue 实例中的 room_list，找到对应的房间实例并更新其目标温度属性,属性的更新将通过 ORM 同步到数据库中，保持应用程序的状态和数据库的数据一致。数据库中的房间记录现在包含了更新后的目标温度。通过调用 Django 的 save 方法，更新后的房间对象会将新的目标温度保存到数据库中

3.1.2 对象设计：消息名称 （比如：对象设计：指令_1(p1,p2,…)）

  1. customer.Customer: 这是系统中的一个客户端实体或角色，触发整个流程。

  2. scheduler.Scheduler: 这个对象负责调度，可能是一个中央控制单元或者服务调度器。

  3. room.Room: 代表一个房间的实体，可能涉及物理位置或者会话环境。

  4. dao.DAO: 数据访问对象（Data Access Object），用于与数据库交互。

  5. server.Server: 服务器对象，可能处理请求并提供服务。

整个交互过程开始于客户端发送一个开启请求（如空调开启请求），包含房间ID和当前房间温度作为参数。

  • 客户端（customer.Customer）发送请求RequestOn(RoomId, CurrentRoomTemp)到调度器（scheduler.Scheduler）。
  • 调度器执行一系列操作，包括获取房间信息GetRoomInfo(RoomId)，创建服务器实例CreateServer()，向服务器添加编号AddServerNum()。
  • 调度器查询是否在房间服务器中InRoomServer(RoomId, CurrentTemp)，这可能涉及到检查当前的服务状态或分配。
  • 之后，调度器对服务器设置各种状态和模式SetServer(ServerState, Mode, TargetTemp, FeeRate, Fee)。
  • 调度器还会设置自己的服务器ID和启动时间SetScheduler(ServerId, ServerStartTime)。
  • 最终，调度器再次确认是否在房间服务器中InRoomServer(RoomId, CurrentTemp)。

同时，还有日志记录的操作：

  • 创建日志条目CreateLog()。
  • 添加日志到数据库AddLog2DB()。

这张图片是一个UML序列图，展示了系统中各个对象之间的交互。以下是根据您给定的格式对图片的描述：

  1. Actor_X（此处为customer:Customer，客户）发出一个指令_1（RequestOn(RoomId, CurrentRoomTemp)，开启请求）。
  2. Object_A（此处为scheduler:Scheduler，调度器）接收到指令_1后，进行以下操作：
    • 通过GetRoomInfo(RoomId)获取房间信息。
    • 使用CreateServer()创建Object_B（此处为serverqueue:ServerQueue，服务器队列）的实例。
    • 使用AddServerNum()增加服务器数量。
    • 使用InRoomServer(RoomId, CurrentRoomTemp)与房间进行交互，包括：
      • 使用BindRoom(RoomId)将房间绑定到服务器。
      • 使用SetServer(ServerState, Mode, TargetTemp, FeeRate, Fee)设置服务器的状态、模式、目标温度、费率和费用。
    • 使用SetScheduler(ServerId, ServerStartTime)设置调度信息。
  3. room:Room（房间）与scheduler:Scheduler（调度器）之间进行交互。
  4. 还涉及到日志记录和数据持久化：
    • log:Log负责通过CreateLog()创建日志。
    • model:Model通过AddLog2DB()将日志添加到数据库。

此序列图提供了客户请求在系统中传递并由各个后端组件处理的视图。通过此图，开发者可以了解在接收到前端请求后，系统内哪些对象将被激活，它们将如何相互作用，以及哪些属性或数据库将被更新。这有助于开发者理解和实施软件后端逻辑。
