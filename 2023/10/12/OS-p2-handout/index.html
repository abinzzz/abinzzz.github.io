Thread的字段：

  • status字段给出了这些状态。(在终止后，将未使用状态赋予Thread。我们在以后的项目中会用到它。)

  • Thread对象还有空间存储一个指向函数的指针(initialFunction字段)和这个函数的参数(initialArgument字段)。这个指针将指向这个线程的“main”函数;当这个线程开始执行时，这个函数会被执行。我们存储了一个指向函数的指针，因为这是一个变量:不同的线程可能执行不同的函数。
  • 我们还可以通过initialArgument字段为这个线程提供一个初始参数。参数必须是整数。通常会有多个线程执行同一个“main”函数。参数是一种方便的方式，可以让每个线程知道它应该扮演什么角色。例如，我们可以创建10个线程，每个线程都使用相同的“main”函数，但向每个线程传递不同的整数(例如，1和10之间)，以让它知道自己是哪个线程。

Thread上执行的操作：

  • 初始化一个新Thread后，我们可以用Fork方法让它开始运行。这不会立即开始线程执行;相反，它使线程准备运行，并将其放在readyList上。readyList是一个Thread链表。readyList上的所有Thread的状态都是准备状态。ReadyList是一个全局变量。还有一个名为currentThread的全局变量，指向当前执行的Thread对象;例如，状态为RUNNING的线程
  • Yield方法只能在当前运行的线程上调用。它将导致切换到其他线程。
  • 仔细阅读Yield中的代码，看看线程切换发生时发生了什么。首先，中断是禁用的;我们不希望线程切换期间有任何干扰。readyList和currentThread是共享变量，当切换线程时，我们希望能够安全地访问和更新它们。然后，Yield会从readyList中找到下一个线程。(如果没有其他线程，那么Yield实际上是。)然后，Yield将使当前运行的进程READY(准备，即不再运行)，并将当前线程添加到readyList的尾部。最后，它将调用Run函数来执行线程切换。
  • Run方法将检查当前线程上是否存在堆栈溢出。然后它将调用Switch来执行实际的切换。

2.5 Switch.s

Switch可能是你遇到过的最迷人的功能了!它位于汇编代码文件Switch.S，你应该仔细看一下。Switch不会返回到调用它的函数。相反，它会切换到另一个线程。然后它返回。因此，返回发生在另一个线程中的另一个函数上!

唯一调用Switch的地方是Run函数，因此Switch返回到其他线程中对Run函数的某些调用。Run的副本(即调用)将返回给调用它的人。这可能是其他Yield的调用，我们将返回另一个Yield它将返回给调用它的人。

Run也会从Sleep中调用，所以我们可能在线程切换后从Sleep调用中返回

第一次创建线程时，一切是如何设置的?当我们有从未调用过它时，我们如何“返回到一个函数”?看一下文件Thread.c中的函数ThreadStartMain和文件Switch.s中的函数ThreadStartUp。

线程终止时会发生什么?看一下Thread.c文件中的ThreadFinish。本质上，
线程处于睡眠状态，永远不可能被唤醒。(难怪他们称之为“线程死亡!”)

接下来，看看当某个线程正在执行时发生定时器中断时会发生什么。这是一个中断，因此CPU首先中断当前例程的执行，并将某些状态压入其(系统)堆栈。然后，它禁用中断并在Runtime.s跳转到名为TimerInterruptHandler的汇编代码例程。它只是调用Thread.c中的TimerInterruptHandler函数。

在TimerInterruptHandler中，我们调用Yield，然后它会切换到另一个线程。稍后，当这个线程再次有机会运行时，我们将回到这里。然后，我们将返回到汇编语言例程，将执行“reti”指令。这将使状态恢复到之前的状态，中断的例程(无论它是什么)将继续。

注意，这段代码维护了一个名为currentInterruptStatus的变量。这是因为查询CPU的“I”位相当困难。更简单的做法是，只要中断状态发生变化，就改变该变量。我们可以在TimerInterruptHandler函数中看到这种情况。显然，中断发生后将立即禁用。Yield数会保持中断状态。因此当我们从Yield返回时，中断将再次被禁用。在返回被中断的线程之前，我们将currentInterruptStatus设置为ENABLED。(它们必须在中断发生之前启用，否则它不可能发生，因此在我们执行“reti”指令之后，状态将恢复到之前的状态，即启用。)

Step 5: Implement the “Mutex” Class

在这一部分中，你必须实现类Mutex。Mutex的类规范在Synch.h中给出了:

你需要为每个方法提供代码。在Synch,c中，您将看到Mutex的behavior构造。有Init、Lock、Unlock和IsHeldByCurrentThread方法，但这些方法有虚拟体。你需要编写这四个方法的代码.

您还需要向Mutex的class规范中添加两个字段，以实现所需的功能.

如何实现Mutex类?仔细看看Semaphore类;您的Mutex的实现将非常相似

首先考虑IsHeldByCurrentThread方法，它可以被任何线程调用。方法的代码需要知道哪个线程持有mutex上的锁;然后它可以将其与currentThread进行比较，看看它们是否相同.因此，您可以考虑向Mutex类添加一个字段(可能是，名为heldBy)，它将是指向持有mutex的线程的指针。当然，需要在锁住mutex时将其设置为当前线程。您可以在该字段中使用null值表示没有线程持有mutex上的锁。

当对mutex请求加锁时，您需要查看是否有任何线程已经对该mutex加了锁。如果是，则需要让当前进程进入睡眠状态。要让线程进入睡眠状态，请查看方法Semaphore.Down。在任何时候，可能有零个、一个或多个线程等待获得mutex上的锁;您需要保存这些线程的列表，以便在执行Unlock时，您可以唤醒其中一个线程。与Semaphore的情况一样，您应该使用FIFO队列，唤醒等待时间最长的线程。

在Lock方法中添加错误检查也是一个好主意，以确保请求锁住mutex的当前线程还没有持有mutex上的锁。如果有，您可以简单地调用FatalError。(这可能表明在使用mutex的代码中存在逻辑错误。它会导致陷入死锁，线程永远冻结，等待自己释放锁。)同样，您还应该添加一个Unlock的检查，以确保当前线程确实持有锁，如果没有，则调用FatalError。稍后你会用到Mutex类，所以这些检查将有助于你在以后的项目中调试。

Main.c提供了TestMutex函数来练习Mutex的实现。它创建了7个线程，这些线程激烈地竞争一个mutex锁。


Step 6: Implement the Producer-Consumer Solution

我们的讲座和一些教科书包含对生产者-消费者问题的讨论，包括解决方案。在KPL中使用类Mutex和Semaphore实现该功能。处理多个生产者和多个消费者，它们共享一个有界缓冲区。

当您尝试完成这个实验时，我们的讲座可能刚刚讨论到使用互斥锁和信号量来正确地实现生产者-消费者问题的解决方案。在这种情况下，如果您希望尽早完成这部分实验，您可能需要在“Three Easy Pieces””教科书中提前阅读一点，从第26章到包括第31.4章(“生产者-消费者问题”)，在讲座覆盖到达这一点之前。如果你不能完全理解解决方案，也没关系，因为我们将在接下来的讲座中详细介绍这个问题.

Mainpackage包含一些用作框架的代码。该缓冲区称为buffer， 包含最多BUFFER_SIZE(例如，5)个字符。有5个生产者进程，每个由线程建模，还有3个消费者进程，每个由线程建模。因此，除了创建其他线程的主线程之外，还有8个线程。
