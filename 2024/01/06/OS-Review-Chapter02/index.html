1.1 进程定义，组成，组织方式

进程定义：进程实体的运行过程，是系统进行资源分配和调度的一个独立单位

与程序的区别：

  • 程序：静态
  • 进程：动态

进行进程切换的时候需要把当前的运行状态记录下来保存在PCB中

进程组成：

  • （1）PCB(进程的管理者所需的数据都放在PCB中)：
    • （1-1）进程描述信息：
      • 进程标识符PID
      • 用户标识符UID
    • （1-2）进程控制和管理信息：
      • 进程当前状态
      • 进程优先级
    • （1-3）资源分配清单
      • 程序段指针
      • 数据段指针
      • 键盘
      • 鼠标
    • （1-4）处理机相关信息：各种寄存器值
  • （2）程序段：代码
  • （3）数据段：数据

组织方式：

  • 链接方式
  • 索引方式


进程的特征：

  • 动态：动态产生，变化，消亡
  • 并发：
  • 独立：
  • 异步：
  • 结构：PCB+程序段+数据段


1.2 进程状态与转换

状态:

  • 运行态
  • 就绪态
  • 阻塞态
  • 创建态: 进程正在被创建，操作系统为进程分配资源、初始化PCB
  • 终止态: 进程正在从系统中撤销，操作系统回收进程拥有的资源、撤销PCB


1.3 进程控制

定义：进程状态的转换，使用原语实现

原语：

  • 使用开关中断来实现(核心态下执行的特权指令)
  • 一气呵成

显然，关/开中断指令的权限非常大，必然是只允许在核心态下执行的特权指令


进程控制相关原语：

  • （1）过程：
    • 更新PCB中的信息：修改进程状态标志，剥夺CPU使用权并保存环境/恢复其运行环境
    • 将PCB插入合适队列
    • 分配/回收资源
  • （2）分类：
    • 创建原语
    • 切换原语
    • 撤销原语
    • 阻塞原语
    • 唤醒原语

1.4 进程通信

进程通信：指进程之间的信息交换

各进程拥有的内存地址空间相互独立，一个进程不能直接访问另一个进程的地址空间

分类：

  • （1）共享存储：互斥访问共享空间
    • 基于数据结构 low
    • 基于存储区 high
  • （2）消息传递：传递结构化信息 （消息头+消息体），两个原语 (发送消息 + 接受消息)
    • 直接通信：放到对方消息队列中
    • 间接通信：先发到中间体
  • （3）管道通信: 设置一个特殊共享文件(管道),其实就是一个缓存区；一个管道智能实现半双工通信；各进程要互斥访问通道；写满不能写，读空不能读；读进程最多只能有一个。


1.5 线程和多线程模型

线程定义：轻量级进程

特性 传统进程机制 引入线程后的机制
资源分配与调度 进程是资源分配和处理机调度的基本单位。 进程是资源分配的基本单位，线程是调度的基本单位。
并发性 只能实现进程间的并发。 支持线程间的并发，提高了并发度。
系统开销 进程间并发需要切换进程运行环境，系统开销较大。 线程间并发，尤其是同一进程内的线程切换，不需要切换整个进程环境，因此系统开销较小。

线程的重要属性：

  • 线程是处理机调度的单位，进程是资源分配的单位
  • 多CPU，各线程可以占有不同的CPU
  • 每个线程有TID和TCB
  • 状态：就绪，阻塞，与你写那个
  • 线程几乎不拥有系统资源
  • 同一进程不同线程间共享进程资源
  • 线程通信无需系统干扰
  • 切换线程比切换进程开销小

线程实现方式：

  • 用户级线程：所有的线程管理工作都由应用程序负责（包括线程切换）
  • 内核级线程：内核级线程的管理工作由操作系统内核完成。
  • 组合方式


2.1 处理机调度的概念层次

调度类型 要做什么 调度发生在 发生频率 对进程状态的影响
高级调度（作业调度） 按照某种规则，从后备队列中选择合适的作业将其调入内存，并为其创建进程 外存 -> 内存（面向作业） 最低 无 -> 创建态 -> 就绪态
中级调度（内存调度） 按照某种规则，从挂起队列中选择合适的进程将其数据调回内存 外存 -> 内存（面向进程） 中等 挂起态 -> 就绪态（阻塞挂起 -> 阻塞态）
低级调度(进程调度) 按照某种规则，从就绪队列中选择一个进程为其分配CPU 内存 -> CPU 最高 就绪态 -> 运行态


挂起与阻塞的区别：

  • 挂起是进程映像在外存
  • 阻塞还在内存

2.2 进程调度的时机, 切换过程, 方式

进程调度： 就绪态 -> 运行态

什么时候需要调度：

  • 主动放弃
    • 正常终止
    • 发生异常终止
    • 主动请求阻塞
  • 被迫中断
    • 时间片用完
    • 更加紧急的事情
    • 更高优先级的进程进入

什么时候不可以调度：

  • 处理中断时
  • OS内核程序临界区
  • 原子操作过程

关于进程在操作系统内核程序临界区中不能进行调度与切换(普通临界区可以进行调度和切换):

  • 临界资源：一个时间段内只允许一个进程使用的资源。各进程需要互斥地访问临界资源。
  • 临界区：访问临界资源的那段代码。


进程调度方式：

  • 非抢占式
  • 抢占式

狭义进程调度与进程切换：

  • 狭义的进程调度： 指的是从就绪队列中选中一个要运行的进程。（这个进程可以是刚刚被暂停执行的进程，也可能是另一个进程，后一种情况就需要进程切换）
  • 进程切"换"：重点在换，是指一个进程让出处理机，由另一个进程占用处理机的过程。

举例说明：


广义的进程调度 = 选择一个进程 + 进程切换


进程切换的过程主要完成了：

  • 对原来运行进程各种数据的保存
  • 对新的进程各种数据的恢复 (程序计数器、程序状态字、各种数据寄存器等处理机现场信息，这些信息一般保存在进程控制块)

注意：进程切换是有代价的，因此如果过于频繁的进行进程调度、切换，必然会使整个系统的效率降低，使系统大部分时间都花在了进程切换上，而真正用于执行进程的时间减少

3.1 进程同步，进程互斥

进程同步(制约关系)

进程同步是为了解决进程的异步问题。

举例说明：
计算1 + 2 x 3,假设系统会产生两个进程：加法和乘法。如果我们想要计算正确，一定要让加法进程在乘法进程之后执行，但由于操作系统的异步性，如果不加以制约，上面操作是无法正确执行的。

异步性：各并发的进程以各自独立、不可预知的速度向前推进。

进程互斥(间接制约关系)

进程互斥指当一个进程访问某临界资源时，另一个想要访问该临界资源的进程必须等待。

临界资源：一个时间段内只允许一个进程使用的资源

临界资源：

  • 进入区：检查是否可以进入，可进入则设置正在访问标志
  • 临界区：访问临界资源
  • 退出区：解除正在访问标志
  • 剩余区：做其他处理

互斥访问原则：

  • 空闲让进：临界区空闲，允许请求进入临界区的进程进入临界区
  • 忙则等待：已有进程进入临界区，其他想进入临界区的进程必须等待
  • 有限等待：对于请求访问临界区的进程，应该保证在有限时间内进入临界区
  • 让权等待：当进程不能进入临界区时，应该立即释放处理机，防止发生"忙则等待"

3.2 进程互斥的软件实现方法

单标志法

思想：两个进程在访问完临界区后会把临界区的权限转移给对方

缺点：如果此时允许进入临界区的进程是p0，而p0一直不访问临界区，那么虽然临界区空闲，但是p1无法访问，违背空闲让进

双标志先检查法

思想：标记各进程想进入临界区的意愿

缺点：如果按照152637顺序执行，p0p1会同时访问临界区，违反忙则等待
原因：检查和上锁不是一气呵成的

双标志后检查法

思想：讲上面方法的检查和上锁互换

缺点：152637的顺序执行两者都无法进入临界区，虽然解决了忙等，但是会产生饥饿现象
原因：同上

Peterson算法

思想：孔融让梨，谁最后退让，对方就先进入临界区

缺点：不满足让权等待

信号量机制的目的：为了更好解决进程互斥和同步的问题

信号量分类：

  • 整型
  • 记录型

一对原语：

  • wait(S) / P
  • sigal(S) / V

整型信号量

信号量和普通整数变量的区别：信号量操作只有初始化、P操作、V操作

通过一个整数型的变量S进行信号量管理，用来表示资源中可供程序使用的数目。例如：某计算机系统中有一台打印机…

缺点：会发生忙等

记录型信号量

目的：为了解决整形信号量的忙等问题

定义一个信号量结构体semaphore，包含一个整型的value用于表示资源数量，和一个指向进程控制块的指针*L。

S.value表示资源的数目
P操作意味着请求资源，所以value–，S.value<0表示资源分配完毕，所以使用block原语进行自我阻塞(运行态->阻塞态)，遵循了让权等待
V操作意味着释放资源，意味着value++，如果S.value<=0表示仍然有进程在等待资源，所以调用wakeup唤醒等待队列中的进程(阻塞态->就绪态)

问题描述：

系统中有一组生产者进程和一组消费者进程，生产者进程每次生产一个产品放入缓冲区，消费者进程每次从缓冲区中取出一个产品并使用。(注: 这里的“产品”理解为某种数据)
生产者、消费者共享一个初始为空、大小为n的缓冲区。
只有缓冲区没满时，生产者才能把产品放入缓冲区，否则必须等待。
只有缓冲区不空时，消费者才能从中取出产品，否则必须等待。
缓冲区是临界资源，各进程必须互斥地访问。

分析：

  • 生产者和消费者对于缓冲区：互斥

3.7 管程和java中实现管程的机制

信号量机制存在的问题：编写程序困难易出错

为了解决信号量的问题，我们提出了管程，可以不需要关注复杂的PV操作

管程的组成及基本特征

管程(class)组成：

  • 局部于管程的共享数据结构(clas中的变量)
  • 对数据结构操作的一组过程(class中的函数)
  • 初始值语句
  • 名字

管程的基本特征：

  • 共享数据结构只能被过程访问
  • 每次进程一个进程在管程里执行过程

4.1 死锁详解

死锁：互相等待对方手里的资源，导致进程都阻塞，都无法向前推进

死锁VS饥饿VS死循环

术语 共同点 区别
死锁 都是进程无法顺利向前推进的现象(故意设计的死循环除外) 死锁一定是“循环等待对方手里的资源”导致的，因此如果有死锁现象，那至少有两个或两个以上的进程同时发生死锁。另外，发生死锁的进程一定处于阻塞态。
饥饿 都是进程无法顺利向前推进的现象(故意设计的死循环除外) 可能只有一个进程发生饥饿。发生饥饿的进程既可能是阻塞态(如长期得不到需要的I/O设备)，也可能是就绪态(长期得不到处理机)。
死循环 都是进程无法顺利向前推进的现象(故意设计的死循环除外) 可能只有一个进程发生死循环。死循环的进程可以上处理机运行(可以是运行态)，只不过无法像期待的那样顺利推进。死锁和饥饿问题是由于操作系统分配资源的策略不合理导致的，而死循环是由代码逻辑的错误导致的。死锁和饥饿是管理者(操作系统)的问题，死循环是被管理者的问题。

死锁的四个必要条件

互斥条件：互斥使用资源的抢夺
不剥夺条件：不能由其他进程强行夺走
请求和保持条件：进程至少保持了一个资源，又提出新的资源请求
循环等待条件：资源的循环等待链

注：死锁一定循环等待，循环等待不一定死锁

发生死锁的时机

  • 各进程对不可剥夺资源的竞争
  • 请求和释放资源的顺序不当
  • 互斥的P是现在同步的P之前

处理死锁的策略

  • 预防死锁：破坏死锁的必要条件
  • 避免死锁：防止系统进入不安全状态
  • 死锁的检测和解除：采取措施接触死锁

处理死锁的策略:预防死锁

破坏互斥条件：例如使用SPOLing技术将打印机改为共享设备
缺点：很多时候无法破坏互斥条件

破坏不可剥夺条件：资源得不到满足就释放，被占有就申请
缺点：实现起来复杂，反复申请和释放会增加开销，可能产生饥饿

破坏请求和保持条件：运行前申请完所有资源，资源未满足不让运行
缺点：资源利用率低，饥饿

破坏循环等待条件：顺序资源分配法
缺点：不方便增加新设备，资源浪费

处理死锁的策略:避免死锁(银行家算法)

安全序列：按照这种序列分配资源，则每个进程都能顺利完成
银行家算法：确保银行在发放现金贷款时候不会发生不能满足所有客户需要的情况
核心思想：资源申请前，先预判此次分配是否会导致系统进入不安全状态

处理死锁的策略:死锁的检测和解除

不预防不避免就会死锁，那我们就来提供几个发生死锁后的解除方法

两种算法：

  • 死锁检测算法：检测系统状态是否发生死锁
  • 死锁解除算法：将系统从死锁状态中解脱出来

死锁的检测：资源分配图法(能够消除所有边则没有死锁发生)

资源分配图组成：

  • 两种结点
    • 进程节点
    • 资源节点(对应一类)
  • 两种边
    • 进程节点->资源节点：申请几个资源
    • 资源节点->进程节点：分配几个资源

死锁的检测

  • 资源剥夺
  • 撤销进程
  • 进程回退
