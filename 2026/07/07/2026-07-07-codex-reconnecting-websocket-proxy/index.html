Codex 每次启动都卡在 Reconnecting？可能是 WebSocket 没走代理

最近折腾 Codex 的时候，遇到一个很烦人的现象：

最迷惑的是，它最后并不是完全不能用。

前面会卡住一会儿，反复重连。等你开始怀疑账号、模型、网络、客户端是不是哪里坏了，它又突然正常开始回答。

这种问题最难受的地方就在这里：不是彻底失败，而是每次都要先罚站一会儿。

现象

典型表现一般是：

  • 打开 Codex 后，第一次提问或者等待模型响应时，会出现多次 Reconnecting...
  • 通常会重试 5 次
  • 每次重试中间都会卡一段时间
  • 重试结束后，Codex 又能继续工作

所以它很容易误导排查方向，让人以为是：

  • 模型响应慢
  • 账号异常
  • OpenAI 服务不稳定
  • Codex 客户端出问题
  • 代理软件没开好

但很多时候，真正的问题可能是：WebSocket 没有正常走代理。

为什么会这样

Codex 默认会优先尝试使用 WebSocket，也就是 WSS 协议，和服务器建立实时连接。

如果 WebSocket 连接成功，体验会更实时。但在一些代理环境里，尤其是终端环境、Windows 代理、Clash、v2rayN 这类组合下，可能会出现一种情况：

HTTP / HTTPS 请求能正常走代理，但 WebSocket 不一定能正常走代理。

方案三：开启代理软件的 TUN 模式

还有一个更彻底的办法，是在代理软件里开启 TUN 模式。

TUN 模式不是只代理某个终端，也不是只靠环境变量，而是从虚拟网卡层面接管系统流量。理论上，Codex 的 WebSocket、HTTPS 以及其他请求，都更容易被代理软件接住。

但我不建议一上来就开 TUN。

它的影响范围比较大，可能会影响：

  • 其他软件联网
  • 公司内网访问
  • 本地开发服务
  • 局域网设备访问
  • 某些 App 的网络行为

也就是说，Codex 可能好了，别的软件又开始出问题。

所以 TUN 更适合作为兜底方案：前两个方案都不行，再考虑它。

建议排查顺序

如果只是想最快恢复使用：

  1. 先用方案一，禁用 WebSocket，强制 Codex 走 HTTP。
  2. 如果想保留默认连接方式，再用方案二，给 Codex 配好代理环境变量。
  3. 如果前两个都不行，再考虑 TUN 模式。

我的倾向是：先用 openai_http 快速验证问题是不是 WebSocket，再决定要不要回头把代理环境变量配完整。
